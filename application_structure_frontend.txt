JANNATBOOKING FRONTEND APPLICATION STRUCTURE
Audited: 2026-06-03

Purpose
This file is a compact frontend reference for future development. It documents the React application layout, route map, role-gated screens, major modules, API clients, and safe-change instructions for the hotel-management and admin experiences.

Frontend Runtime
- hotels_frontend is a Create React App application using React 18, react-router-dom v5, Ant Design v5, styled-components, axios, and domain-specific UI modules.
- src/App.js is the main route registry.
- src/auth/AdminRoute.js protects platform admin routes.
- src/auth/HotelRoute.js protects hotel-management routes and hydrates selectedHotel context for legacy hotel screens.
- Most API calls are grouped in:
  - src/AdminModule/apiAdmin.js
  - src/HotelModule/apiAdmin.js

Saudi Date And Hijri Formatting
- src/utils/saudiDates.js is the shared formatter for PMS-facing dates.
- It formats all Arabic PMS dates against Asia/Riyadh and uses the Umm al-Qura Hijri calendar through Intl locale ar-SA-u-ca-islamic-umalqura-nu-latn.
- Navbar clock/date, overall reservation/financial/account/housekeeping tables, reservation notifications, reservation detail modals, and overall reservation Excel exports should use this helper instead of hand-written moment(...).format("DD/MM/YYYY") or browser-default toLocaleDateString().
- Arabic screens should prefer readable Saudi Arabic dates such as "24 مايو 2026 | 7 ذو الحجة 1447 هـ" where space allows. Dense inputs and backend payloads should remain ISO yyyy-MM-dd.
- Keep Latin digits for PMS consistency unless a screen is intentionally redesigned for Arabic-Indic numerals.

Frontend Route Guards And Role Behavior
- src/auth/AdminRoute.js
  - Requires a signed-in active admin.
  - Allows role 1000, configured super-admin users, and users whose normalized role descriptions represent superadmin/super admin.
  - Wraps admin screens with Ant Design ConfigProvider and popup-container handling.
- src/auth/HotelRoute.js
  - Builds user permissions from role, roles, roleDescription, roleDescriptions, and accessTo.
  - Treats role 10000 or roleDescription systemadmin as Hotel System Admin access, scoped by assigned hotels.
  - Allows root owners and admins broad hotel-management access.
  - Platform super admins must remain global in hotel-management routes even if their account has limited or stale hotel assignment arrays.
  - Allows limited order takers only to the new-reservation/list-style flows they are allowed to use.
  - Hydrates localStorage.selectedHotel from the route hotel id so legacy hotel screens keep working.
  - Allows inactive public applicants with applicationReview.status pending to see only a pending-review dashboard message.

Role Expectations In The Frontend
- Super admin / role 1000
  - AdminModule and hotel-management access across hotels.
  - Also recognize configured super-admin users and normalized role descriptions like superadmin / super admin as the same global platform role.
  - UI should not hide account-management, staff-management, approval, or hotel-management actions from a true super admin because of hotelIdWork, hotelIdsWork, hotelIdsOwner, hotelsToSupport, or belongsToId.
  - Overall-dashboard account update screens may send Hotel System Admin updates without ownerId query params; the backend is responsible for validating selected hotel IDs globally for super admins.
- Root owner / role 2000 without belongsToId
  - Owner-level hotel-management and overall-dashboard access for owned hotels.
- Hotel System Admin / role 10000 or systemadmin
  - Hotel-side employee/admin for one hotel or a group of hotels.
  - Owner-like operational access but limited to assigned hotels.
  - Keep the internal roleDescription as systemadmin for compatibility; user-facing labels should say Hotel System Admin.
- Hotel manager / hotelmanager
  - Hotel-management access for assigned hotel scope, including operations, settings, staff, reservations, housekeeping, and reports where authorized.
- Reception / role 3000 or reception
  - New reservation and reservation-list workflows for assigned hotels.
- Finance / role 6000 or finance
  - Financial reports, wallet/financial actions, selected reservation reports, and finance-facing dashboard areas.
- Order taker / role 7000 or ordertaker
  - Limited new-reservation and own-reservations access. Booking source should default to agent/company identity.
  - Public agent applicants remain inactive and pending until a selected hotel approves them.
- Reservation employee / role 8000 or reservationemployee
  - Reservation operations, pending confirmations, selected settings, and reservation-related reports.
- Housekeeping manager / role 4000 or housekeepingmanager
  - Housekeeping reports, assignment, updates, and operational task management.
- Housekeeper / role 5000 or housekeeping
  - Assigned housekeeping tasks and limited housekeeping task visibility.

  - The account list is split into Hotel Employees and External Agents tabs. The employee tab excludes order-taker/agent rows; the agent tab shows only external agents. Keep table filters and backend role params aligned with the selected tab. Opening update-account must preserve/set accountView from the account role so the editor lands on the matching staff/agent tab.

  - The generate-link action follows the active tab: staff mode opens a staff invitation that requires an employee role, while agent mode opens an agent invitation with agent commercial settings. Both modes still require explicit hotel selection.

- src/HotelModule/TheOverallStructure/OverallAccountManagement/UpdateExistingAccount.js
  - The update-account route mirrors account-management with Hotel Employees and External Agents tabs driven by the shared accountView query param. Each tab filters the account list and role controls to its own account type, and accountId may preselect the matching row when it is present in the active tab.
- src/HotelModule/TheOverallStructure/OverallReservationsList/
  - Multi-hotel reservation overall views, pending reservations, and the embedded overall new-reservation flow.
  - Overall reservation tables use the shared OverallTableWrap/ReservationTableControls styling, 25-character tooltip truncation, Gregorian/Hijri date toggle with Gregorian as the default, and sort buttons for source, hotel, arrival, and departure.
  - Reservation table rows should keep nights and price-per-day columns visible in both the regular reservation list and pending-reservations list, and Excel export should preserve those metrics.
  - OverallReservationMain uses a compact search/filter row, then a clickable scorecard strip between filters and sorting. Scorecard filters must stay query/backend aligned and should not inflate button height or whitespace.
  - The reservation-main table column order follows the operational screenshot: reservation date, hotel, confirmation, guest, highlighted financial metrics, status, source, arrival, departure, availability, payment, paid amount, and row actions/details. The nights, price-per-night, and total columns are visually highlighted while staying readable.
  - Overall reservation detail modals reuse ReservationDetail and should keep the hotel name visible in the center action card above the update button so users know which hotel owns the selected reservation.
  - ReservationDetail's top header uses three primary cards plus a secondary action strip. Keep edit-reservation centered in the strip, not inside the lifecycle card, so the lifecycle card stays focused on status progress.
  - OverallPendingReservations is read-only for approved order-taker/agent accounts. Agents can see their own pending, rejected, finance-review, and commission-review reservations across assigned hotels, but confirmation-number clicks stay inside the overall detail modal instead of navigating into hotel-staff pending workflows.
- src/HotelModule/NewReservation/
  - Single-hotel reservation creation, guest/reservation forms, room selection, calendar/heat-map, pending confirmations, and reservation imports.
  - Important files include NewReservationMain.js, ZReservationForm.js, ZReservationForm2.js, HotelHeatMap.js, HotelRunnerReservationList.js, and PendingConfirmationReport.js.
  - Agent/order-taker availability requests pass the current agent id so assigned stock is prioritized, and room chips show assigned stock when an override applies.
  - Agent-scoped availability now treats pending-confirmation agent reservations as consuming that agent's assigned stock. This keeps the room chips aligned with backend enforcement: stock=1 becomes available=0 after the first pending agent reservation, and stock=0 blocks all agent reservation attempts for that room.
  - PendingConfirmationReport.js must show reservations whose backend pendingConfirmation.status is "pending" even when reservation_status remains "confirmed" for client-visible public/paid bookings.
- src/HotelModule/HouseKeeping/
  - Housekeeping task list, task details, assignment/update flows, and supplies.
- src/HotelModule/HotelSettings/
  - Hotel setup and configuration: details, rooms, photos, location, payment settings, OTA/payment/business setup, and completion state.
  - ZUpdateCase1.js should mirror the clean add-room details layout from ZCase1.js: English and Arabic name/description fields are inline, not hidden behind separate black modal buttons, while edits still update the selected existing room object.
  - In both ZCase1.js and ZUpdateCase1.js, keep Active/Inactive and the agent stock/pricing action on the same row but in separate framed controls so the agent assignment action reads as independent functionality.
  - ZCase3.js and ZUpdateCase3.js custom-calendar room pricing must show both regular/range price and root price to authorized room editors in add and update flows. The root price is not super-admin-only inside the room settings workflow.
  - ZAgentRoomOverridesButton.js adds the wide room-level modal for authorized users to assign private agent stock and agent-specific calendar pricing while adding or updating a room. It uses the lightweight /hotel-agents option endpoint with server-side search/limit instead of loading full staff records. On existing rooms, the modal save actions call the narrow agent-overrides PATCH endpoint and persist only that override portion; on brand-new unsaved rooms, the data stays local until the room itself is saved. The Pricing sub-tab supports a blocked date-range toggle that saves private agent-only blocked rows, shown as Blocked/Mahzor-style calendar/table entries, so only that agent is prevented from booking overlapping dates. The modal should remain a roomy PMS-style workspace, with Stock/Pricing rendered as polished top sub-tabs, and the pricing calendar must support drag-selecting a date range that syncs into the pricing range picker. Agent pricing calendar event labels are rendered on stacked lines, e.g. Price on one line and Root price on the next, so values remain readable in month cells.
- src/HotelModule/HotelStaff/
  - Staff account creation and staff-role forms.
- src/HotelModule/Financials/
  - Hotel financial dashboards, manager financial modal, wallet/agent financial operations.
  - In the overall financial report/wallet screens, pure order-taker/agent accounts are pinned to their own agent id even if query params are manually changed. They may see assigned-hotel financial data only for their own wallet, reservations, commissions, and claims.
  - Overall financial report, financial actions, and wallet-management menu entries are visible to order-taker/agent users, but the backend and frontend must keep those users scoped to their own agent records only.
  - Agent wallets are global per agent, not per hotel. OverallWalletManagement must not ask for a hotel when adding wallet money or submitting an agent wallet claim; the backend derives authorized hotel scope from the logged-in actor and the selected/global agent.
  - OverallWalletManagement uses a compact page title instead of the old "new movement" heading. Arabic labels may remain Arabic, but numeric figures in scorecards should render with Latin/English digits for PMS consistency.
  - OverallWalletManagement is a two-tab workflow: Add new creates fresh manual wallet movements/agent claims, while Update loads an existing transaction by table action or transactionId query param. Finance wallet notifications must route to overall=wallet-management&walletTab=update with agentId and transactionId so the selected movement opens in update mode. The save action in the Add/Update form should remain centered and visually dominant so users understand the movement is submitted only through that primary button.
  - Overall financial report and manager financial modal should fetch the agent wallet summary once from the global wallet endpoint, not once per hotel, so wallet movements are not duplicated across assigned hotels.
  - Single-hotel agent wallet ledgers must show rejected wallet-claim reasons clearly in the transactions table and exports, because the notification may be acknowledged before the agent reviews the financial page.

  - Open overall=financial-report, overall=financial-actions, and overall=wallet-management. The UI must stay pinned to the logged-in agent even if agentId is manually changed in the URL, and the visible rows must be only that agent's wallet, reservation, commission, and claim data.
  - Repeat as reservation employee, finance, manager, Hotel System Admin, and super admin to confirm their broader menu/table access still works normally.

Overall Dashboard Menu Rules
- Owner-like users see the complete overall dashboard:
  - super admin, role 1000, root owner role 2000, and Hotel System Admin role 10000/systemadmin within assigned hotel scope.
- Summary is visible to hotelmanager, finance, reservationemployee, housekeepingmanager, order-taker/agent users, and owner-like users.
- The Hotels Summary tab is the default landing page for hotel-side users after sign-in. It embeds the former My Summary reservation-department report above the existing hotels summary table. The embedded report reuses the actor's overall hotel scope, so agents see only their own/assigned-hotel reservation data while finance, reservation employees, managers, owners, Hotel System Admins, and super admins see their authorized scope.
- New reservation is visible to reception, ordertaker, reservationemployee, hotelmanager, and users with accessTo newReservation.
- Reservation lists are visible to reception, finance, ordertaker, reservationemployee, hotelmanager, and owner-like users.
- Pending reservations are visible to finance, reservationemployee, hotelmanager, and owner-like users.
- Housekeeping is visible to housekeepingmanager, housekeeping, hotelmanager, and owner-like users.
- Financial report/actions/wallet views are visible to finance, hotelmanager, owner-like users, and order-taker/agent users scoped to their own records only.
- Hotel map is visible to reception, finance, reservationemployee, hotelmanager, and owner-like users.
- Settings are visible to reservationemployee, hotelmanager, and owner-like users.
- Account Management sublinks are hidden for non-owner-like users.

Important UI And UX Notes
- The application is Arabic/RTL-heavy but still contains English labels and LTR values. Components that show names, emails, dates, confirmation numbers, and hotel labels must handle mixed direction safely.
- Keep mobile behavior first-class for overall-dashboard filters, reservation forms, tables, admin modals, and side menus.
- Ant Design dropdowns inside modals must have a popup container/z-index that keeps option clicks usable over the modal mask.
- Tables should avoid horizontal data collisions by using controlled widths, wrapping, ellipsis/tooltips where appropriate, and responsive card/table alternatives when needed.
- The overall new-reservation hotel selector is intentionally prominent before hotel selection and should stop attention animation on hover, focus, pointer down, keyboard interaction, or selection.

Frontend Critical Workflows To Protect
- Public checkout and client-visible reservation status
  - Public/client-created reservations are internally pending hotel confirmation, including paid PayPal bookings, chatbot-assisted bookings, and verified public bookings.
  - Paid guests must continue to see calm confirmed/created messaging after successful payment. Frontend public checkout, payment-link, invoice, WhatsApp-triggering, and single-reservation screens should not display internal pending-confirmation wording.
  - Backend public responses remove pendingConfirmation/agentDecisionSnapshot and apply pendingConfirmation.clientVisibleStatus when needed. Do not reintroduce UI that reads internal hotel-review fields on public pages.
- Public signup
  - /signup must keep the existing owner flow unchanged when Hotel Manager / Owner is selected.
  - Without query params, Hotel Manager / Owner is selected by default.
  - /signup supports short invitation links generated as /signup?invite=<code>. The page resolves the code through GET /api/signup-invitation/:code, then locks the signed hotels, application type, role, agent settings, and optional prefill data.
  - Legacy /signup?invitationToken=<jwt> links remain supported for backwards compatibility, but new links should use the short invite code.
  - Invited hotel selections must display hotel names and optional location labels, not raw hotel ObjectIds.
  - Only hotel managers/owners can apply directly from /signup without a link.
  - Agent and job/employee signup are invitation-only. Without a valid invitation code/token, the public signup page must show a friendly message directing the applicant to hotel management for the correct link.
  - Invited agent/employee signup is locked to the verified invitation. The applicant can complete personal details and set password; hotel, role, and agent settings stay locked.
  - Pending applicants should not reach reservation, finance, settings, or housekeeping workflows until approved.
  - The pending applicant dashboard should show a clear Arabic pending message, refresh approval status immediately, poll periodically, and move the user into the dashboard after approval.
  - Attachments are optional and are sent as companyDocuments for reviewer visibility.
- Overall create-account invitations
  - /hotel-management/main-dashboard?overall=create-account is a two-tab HR workflow: Internal Staff and External Agents.
  - Staff mode intentionally starts with no preselected hotels and no preselected staff roles. It excludes the external-agent/order-taker role from the role grid, and account creation is blocked until hotel, role, identity, and password requirements are met.
  - Agent mode intentionally starts with no preselected hotels but locks the account role to ordertaker. It shows the agent business model and opening-wallet fields, then submits the correct agent role/access payload without asking HR to choose an agent role manually.
  - The generate-link button follows the selected tab. Staff mode creates a job/employee invitation and still requires an employee role; agent mode creates an agent invitation and may include agent commercial model/opening wallet data. The modal should not allow users to switch the invitation type away from the active tab.
  - The modal sends hotels assigned to the manager/owner, the locked application type, employee role or agent commercial model, and optional prefilled contact/company data to the backend.
  - New invitation links use a short /signup?invite=<code> URL. The backend stores the invitation payload securely and returns the code; frontend copy/email/WhatsApp actions should only enable after the link is generated.
  - Backend public signup validation and pending approval still control final access.
- Overall new reservation
  - Require hotel selection from the actor's authorized hotels.
  - Pass selected hotel id and owner id into the embedded NewReservationMain flow.
  - Keep room inventory and room selection tied to the selected hotel, not stale localStorage.
  - For agents/order takers, source defaults to agent/company identity. Hotel staff should provide a proper booking source.
- Single-hotel new reservation
  - Route /hotel-management/new-reservation/:userId/:hotelId must remain the reference behavior for the embedded overall new-reservation component.
- Housekeeping
  - Authorized managers can view task details, assign/update tasks, and notify housekeepers.
  - Housekeepers should only interact with assigned/authorized tasks.
- Reservation and housekeeping status
  - Do not add UI paths that allow non-super-admin users to move closed/completed status backward.
  - UI should reflect backend restrictions instead of hiding backend failures.
- Account management
  - Account creation/update screens must only show hotels the acting user can manage.
  - /hotel-management/main-dashboard?overall=account-management is split into Hotel Employees and External Agents tabs. The employee tab must not show order-taker/agent accounts, and the agent tab must not show internal hotel staff.
  - For platform super admins, "can manage" means all relevant hotels and accounts. Do not apply scoped staff limitations to role 1000 or superadmin/super admin users.
  - Hotel System Admin saves use the legacy /api/overall-dashboard/system-admin/:accountId/:userId endpoint. Super admins can assign any existing hotel; scoped users can only assign hotels returned by their allowed overall account context.
  - Role selection treats Hotel System Admin as mutually exclusive. Selecting Hotel System Admin must immediately clear all other roles and disable them; unchecking Hotel System Admin must reopen the rest of the internal staff role options. Agent creation is handled by the External Agents tab, not by mixing ordertaker into staff role selection.
  - In update-account, endpoint selection follows the currently selected roles, not the original account role. If Hotel System Admin is selected, use the system-admin endpoint; if it is removed, use the hotel-staff update endpoint to convert the account back into a scoped hotel role.
  - Successful account create/update responses are backed by backend ActivityTracker records. Frontend should keep sending the complete role, hotel scope, accessTo, active state, and relevant agent/company fields so backend audit entries are meaningful.
  - /hotel-management/main-dashboard?overall=update-account uses the same Hotel Employees / External Agents accountView tabs as account-management. The editor may preselect the accountId row when that account is present in the active tab; otherwise it should show the language-aware prompt until the user clicks a row.
